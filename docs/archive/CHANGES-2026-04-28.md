## 2. 与 DESIGN.md 的契约偏差（需要核验）

仅一项。其余实现严格按 §3.x 落地。

**`CommandContext.listCommands(): readonly CommandHandler[]`**

- **背景**：F5 要求「内置 `/help` 列出已注册命令」，但 §3.5 原本的 `CommandContext` 不暴露 registry。`/help` 需要拿到当前命令列表才能渲染。
- **取舍**：相比把 `/help` 特例化（违反「命令统一插件化」/AC-11），更小的改动是给 ctx 加一个只读列表访问器。
- **影响**：插件作者可以读这个列表（理论上能写自定义 help），不能改它（`readonly`）。不影响其它 AC。
- **已落实位置**：`src/plugins/api.ts`、`src/router/dispatch.ts`、`src/commands/help.ts`、`docs/DESIGN.md` §3.5。

如果你不接受这个偏差，备选方案是把 help 从插件目录里拿出来作为内置命令，但这会破坏「新增/修改命令文件无需重启」对 help 的对称性。建议保留。

## 3. 关键实现细节（容易踩坑的地方）

### 3.1 群消息触发：剥 reply / at / 前导空白

`router/trigger.ts::stripLeadingAtAndReply` 处理顺序：
1. 抛掉前导 `reply`（QQ 客户端在引用回复时常带）。
2. 抛掉第一个 `at` 且 `qq === selfId` 的段。
3. 抛掉夹在 at 前后的纯空白 text 段。
4. 对剩余段的首段 text 做 `replace(/^\s+/, '')`。

只有这一系列处理之后还以 `prefix` 开头才触发。AC-5/6/7/8 都靠这套规则。

### 3.2 限流是「令牌桶 + warned 一次」

`router/dispatch.ts::checkRateLimit` 每个 `userId` 一份状态：
- `times[]`：滑动窗内的触发时间戳。
- `warnedUntil`：下一次允许再发警告的时间。

行为：
- 窗内 < `perUser` 次：放行。
- 第一次超额：回 `操作过于频繁，请稍后再试`，记录 `warnedUntil = now + windowMs`。
- 后续仍在 `warnedUntil` 内的超额：完全静默。
- 窗口滑出后：状态自然恢复。

AC-12 用 7 连发验证序列：`[ok×5, warn, silent]`。

### 3.3 命令热加载靠 ESM 查询参数

Node ESM 没法清缓存，但 `import('file:///...?v=12345')` 每个 url 都被当成独立模块，所以每次 `loadFile` 都附 `?v=Date.now()`。`registry.ts::loadFile`。

### 3.4 配置热加载语义

`config/loader.ts` 把 `listen.{host,port,token}` 的变更标记为 `restart-required` 并发出告警，其它字段直接 emit `change`，订阅方（`index.ts`）刷新 logger level，dispatcher 通过 `() => cfg` 闭包总是读最新值。

### 3.5 安全 / 日志承诺

按 CLAUDE.md「Logging & safety」：
- token 不写日志（只在 401 时记 peer addr）。
- 用户消息内容不写 info；只 `mid + cmd + metadata`。
- handler throw 走 dispatch try/catch，回 `命令执行失败`，stack 入 error 日志。

人核验关键点：搜 `log.info` 和 `log.error`，确认没漏带 `text` / `raw_message` / `token`。已检过一遍，没有。

均已同步落到 DESIGN.md，无未声明偏差。`CommandContext` 新增 `llm: LlmClient` 在 §3.5 里写明；`cfg.llm` 在 §3.4 示例里出现。

**硬规则**（写进 §3.5、CLAUDE.md「Current state」、本文）

- LLM 是 handler 作者用的基础设施，**不向终端用户暴露任何直接对话入口**。
- 不允许 `/ask`、`/chat` 之类的对话型命令。`/translate` 是有界工具命令（输入 → 中文译文），不是对话入口。

**日志承诺**（与现有 §3.5 "Logging & safety" 一致延伸）

- 成功调用 info 级一行：`{ provider, model, latencyMs, msgCount, promptTokens, completionTokens, finishReason }`。
- 失败 warn 级：`{ provider, model, code, httpStatus?, latencyMs }`。
- HTTP 4xx/5xx 的 body snippet 仅 debug 级（避免上游回显密钥时漏到 info）。
- **绝不**记 `messages` 内容、响应文本、图片 URL、`apiKey`。

**自动化校验**

- `pnpm typecheck` —— 0 错误。
- `pnpm smoke` —— **13/13 通过**。LLM 改动未触碰路由 / 触发 / 限流，行为零变化。`/translate` 因依赖外部 API 不进 smoke。

**仍需人工核验**

1. **`/translate` 文本路径**：配 `cfg.llm.default` + 一个真实 provider，私聊 `/translate Hello`，应收到中文译文。日志一行 `llm chat ok` 带 token 数，**不含**任何用户内容或密钥。
2. **`/translate` 图片路径**：配视觉模型（如 `gpt-4o-mini` / `qwen-vl-*`），`/translate` 同时附图，应收到图中文字的中文译文。
3. **未配置错误**：注释掉 `llm` 段，`/translate hello` 应回 `'翻译功能未配置（管理员需在 bot.json5 设置 llm）'`。
4. **错误密钥**：把 apiKey 改错，应回 `'命令执行失败'`，error 日志看到 `LlmError('HTTP', ..., 401)`，**不**含 apiKey。
5. **热加载**：bot 跑着改 `cfg.llm.providers.<name>.model`，下次 `/translate` 调用日志里 model 字段应为新值，无需重启。
6. **空输入**：`/translate`（无 argv 无图）→ 用法提示，**不**调 LLM（日志里没有 `llm chat ok`/`llm http error` 等行）。

**`/translate` v1 限制**

- 只翻中文，不做语言识别 / 目标语言切换。
- 一次调用，一次回复，长文不分片（撞 QQ 长度上限再补）。
- 不维护历史。
- 默认 provider 必须支持视觉，否则带图调用会回 `'命令执行失败'`（provider 返回 400/422）。这是配置问题，不是 bot 问题。

**为什么没有 `/ask`**

用户原话："不可以让用户直接和LLM产生对话"。设计上 LLM 必须是命令的"工具"而非"对话窗口"。`/translate` 是有限定输入的工具命令；如果以后加 `/define <word>`、`/explain <code>` 之类，都遵循同一模式。`/ask <free-form>` 类命令实质就是聊天入口，禁用。
