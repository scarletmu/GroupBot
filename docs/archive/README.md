# Archive 归档目录

> 这个目录存放**已封存的历史文档**。日常开发不读它，AI 代理也不要主动读。它的存在是为了把"当时的决策与理由"、"过去某天发生了什么"留下记录，万一以后要追溯能找回来。

如果当前活动文档（`docs/architecture.md` / `config.md` / `plugins.md` / `acceptance.md`）和归档里的某份文档说法不一致，**以活动文档为准**——归档的内容是冻结在写下时刻的快照，不会随项目演进更新。

## 当前归档清单

- **`DESIGN.md`** — 项目最初的整篇设计稿。P0+P1 阶段（2026-04-28 之前）作为唯一 source-of-truth 用过。代码全部落地、LLM 共享客户端加上之后，被拆成了 `docs/architecture.md` / `config.md` / `plugins.md` / `acceptance.md` 四份活动文档，原稿移到这里封存。如果想看"为什么当时选 NapCat"、"为什么用反向 WS"、"为什么不引入框架"这些历史决策，可以读这一份。
- **`CHANGES-2026-04-28.md`** — 2026-04-28 这一天的工作日志，记录了 P0+P1 + LLM 共享客户端 + `/translate` 命令的全部交付清单与契约偏差。属于一次性记录，不维护。

## 什么时候归档

某份文档应该挪进 `archive/` 的信号：

- **被替代**：内容被新的活动文档拆分/重写，原稿没必要再维护。挪进来的同时记得在 `docs/` 下相应位置写好新文档，并更新 `CLAUDE.md` 索引。
- **属于一次性记录**：比如某天的会议纪要、某次发版的 changelog、某次重构的 PR 描述备份。这些天然只代表当时的状态，不需要更新。
- **`docs/plans/` 里的提案落地或取消了**：实施完毕的提案挪过来，作为"当时为什么这么做"的可追溯记录；取消的提案也挪过来，作为"我们考虑过这个方向但放弃了"的备忘。

## 归档时的规矩

1. **直接 move，不要 copy**——避免活/封两份长得像但又不完全一样的文档。
2. **顶部加一行注解**：写明归档原因和归档日期，比如：
   ```
   > Archived 2026-04-28. Replaced by docs/architecture.md / docs/config.md / ...
   ```
   或：
   ```
   > Landed 2026-04-28. Implementation in: src/llm/, src/commands/translate.ts.
   ```
3. **更新所有外部链接**：`README.md` / `CLAUDE.md` / 其它活动文档里如果引用了这份文档，要么把链接改到新位置，要么（更常见）把链接干脆删掉，因为活动文档已经覆盖。
4. **更新本 README 的清单**：在上面"当前归档清单"里加一条，一句话说明它是什么、为什么进来。

## 不要做的事

- **不要在归档文档里改内容**：哪怕发现了错别字也别改。归档就是冻结状态。要纠错请在活动文档里写正确版本。
- **不要把活动开发产物丢进来**：临时草稿、半成品、还在用的笔记不要进，那些应该在 `docs/plans/` 或别的地方。
- **不要让 AI 代理把这里当成工作面文档去读**：CLAUDE.md 已明确告诉它这里属于 Tier 3，仅在用户明确点名时才读。
