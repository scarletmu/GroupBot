# Plans 目录说明

> 这份说明写给**人类**——准备给项目加新功能的人。AI 代理不需要主动读这个目录，除非用户明确指向某一份具体的提案文件。

## 这是干嘛的

这个目录用来放**还没实施、但已经成型到可以讨论**的功能提案。每个文件就是一份小型设计稿：写清楚要做什么、为什么做、改动哪些文件、有哪些待定问题。

为什么要走这一步：

- AI 代理的优势是**按既定方向高效实施**，劣势是**容易在不明确的目标下越权设计**。先把"我们要做的边界、不做的边界"用文字钉死，再让 AI 去实施，能少踩很多坑。
- 一份提案可以让你在动手之前自己想清楚（写下来才会发现哪里没想透）。
- 多份提案排队的时候，可以一眼看到哪些值得先做、哪些其实可以合并、哪些可以放弃。
- 提案落地之后挪进 `../archive/`，就成了"为什么当初这么做"的可追溯记录。

## 一份提案的结构

不是模板，是建议格式，自己看着改：

```markdown
# <功能名>

**Status:** proposed | in-progress | landed (→ archive/<file>) | dropped
**Owner:** （谁负责，可以是你自己）
**Created:** YYYY-MM-DD

## Context（为什么要做这个）
解决什么问题？谁会受益？不做会怎样？为什么是现在做？

## Proposal（要做什么）
改动的形状。具体加什么文件、改什么文件、新增什么配置字段、新增什么命令。
关键文件路径列出来。如果有契约（接口、数据结构），把契约写进来。

## Boundaries（边界）
明确**不做**什么。最容易出问题的就是这一节没写——AI 实施时会自动扩张到模糊地带。

## Open questions（待定）
还没想清楚的点。讨论清楚之前不要让 AI 实施。

## Verification（怎么验收）
功能怎么算"做完了"？有没有可以加进 `pnpm smoke` 的自动用例？需要人工测哪些场景？

## Risks（风险）
可能踩的雷。最常见的：影响触发规则、影响热加载、影响日志安全、引入新依赖。
```

## 工作流

1. **起草**：在这里建一个 `<功能>.md` 写出来。
2. **讨论**：跟自己/合作者过一遍 `Boundaries` 和 `Open questions`，把待定项收掉。
3. **实施**：把这份文件丢给 AI 让它按上面的 Proposal 实施。AI 可以读这份具体文件，但不会读这个目录里其它文件。
4. **归档**：实施完成后，**把这个文件挪到 `../archive/`**，并在文件顶部加一行：
   ```
   > Landed YYYY-MM-DD. Implementation in: <相关文件路径>。
   ```
5. **取消**：如果决定不做，加 `Status: dropped` 和一段为什么取消的说明，然后挪去 `../archive/`，便于将来回看为什么放弃。
